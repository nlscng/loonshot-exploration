## Session Summary

### Tuesday Phase: Identify Barriers — Meta-runner v2

Reviewed the meta-runner v1 codebase in detail and identified 8 concrete barriers across the 3 session deliverables. Also evaluated whether MCP server integration fits Cycle 2 — concluded it doesn't (yet).

#### Session 1 Barriers: Memory (SM-2 + `memory.json`)

**Barrier 1: No persistence hook.** The CLI loop is `input() → handle_message() → print()` with no save/load lifecycle. Need to decide *when* memory writes happen — every answer? On quit? Ctrl+C exits lose data.

**Barrier 2: `session_cards_seen` initialized but never used.** Quiz currently allows repeats within a session. SM-2 needs `(card_code, quiz_type)` pair tracking — the existing set only tracks card codes. Need a design decision on tracking granularity.

**Barrier 3: Card selection is random.** `_generate_question()` picks quiz types randomly (60% meta-biased). SM-2 needs to *replace* this with priority-based picking (overdue → weak → unseen → random reinforcement). This rewires the core quiz loop — deeper than "add a JSON file."

#### Session 2 Barriers: Planning (`plan_session()` + opening message)

**Barrier 4: No session boundary concept.** The CLI loop has no "session start" or "session end" — it's `while True`. Need to inject a planning step before the first question. `on_members_added_activity` is the SDK hook for this but isn't implemented — the bot runs as CLI only.

**Barrier 5: Cold start problem.** `plan_session()` depends on card memory (overdue reviews, weak areas). First session ever has no history → planner has nothing to plan from. Need a cold-start strategy (default to meta staples? random selection with explanation?).

#### Session 3 Barriers: Initiative (unprompted commentary + course correction)

**Barrier 6: `handle_message()` returns a single string.** Initiative means extra commentary — "That's 3 in a row" or "Switching topics." Current architecture is `str in → str out`. Need either multi-part responses or a response builder that appends commentary to quiz results.

**Barrier 7: No rolling performance tracking.** Agent tracks `correct/wrong` totals but not streaks, per-topic accuracy, or recent trend. Mid-session course correction ("3 wrong in a row — switching") requires a rolling window, not just cumulative counts.

#### Cross-cutting

**Barrier 8: Test fixtures bypass `__init__`.** Tests use `patch.object(QuizAgent, "__init__", lambda self: None)` and construct agents manually. Adding memory changes the init path (load JSON, set up SM-2 state). Existing fixtures will need updates or they'll silently skip new behavior.

### MCP Server Evaluation

Evaluated whether to incorporate MCP server into Cycle 2. Conclusion: **not now.**

- Own Cycle 1 analysis ranked MCP #8 of 11, labeled "disqualifying" — no behavioral test exists
- Devil's advocate called it "technology in search of a problem"
- MCP fits as a Track B / Phase 3+ addition or Cycle 3 project — after agent behaviors are proven
- Tightest scope if pursued later: finish Track A (sessions 1-3), then session 4 = wrap working SM-2 + planner as MCP tools

### Hardest Barrier

**Barrier #3 (rewiring card selection)** is the critical path. Everything else is additive. If SM-2-driven card selection works, sessions 2 and 3 become straightforward extensions of the same priority queue.
