## Meta-Runner Retrospective

### The Original Vision

From the Feb 23 framing:
- **Who:** Myself and other Netrunner players learning the standard card pool
- **What:** An agentic app that teaches cards through quizzes, tracks progress, adapts difficulty
- **Primary goal:** Learn to build an agent — the Netrunner theme keeps it fun and bounded
- **Secondary goal:** Actually useful for learning cards

The key word in the framing was *agentic* — the agent should take initiative, plan sessions, and adapt autonomously.

### What Was Built

| Component | Status | Notes |
|-----------|--------|-------|
| 1. Card Data Layer | ✅ Done | 2,460 cards from NRDB v2 API → SQLite |
| 2. Agent Skeleton | ✅ Done | MS Agents SDK ActivityHandler, CLI |
| 3. Quiz Tool | ✅ Done | 6 quiz types including meta-aware questions |
| 7. Meta Data Layer | ✅ Done | 1,239 decklists, 1,879 cards with popularity stats |
| 4. Learning Memory | ❌ Not built | No persistence between sessions |
| 5. Session Planner | ❌ Not built | No autonomous session planning |
| 6. Card Lookup Tool | ❌ Not built | No on-demand card info outside quiz |

Code quality is solid: 639 lines of source, 667 lines of tests (67 tests, 100% pass), clean separation between data layer and agent logic, proper SQLite schema with indexes.

### Success Metrics (from Mar 4 journal)

| Metric | Target | Result |
|--------|--------|--------|
| Agent runs locally | Starts, accepts input, responds | ✅ Achieved |
| Card data loaded | ≥90% of standard pool cached | ✅ Achieved (2,460 cards) |
| Quiz works | Valid questions with correct answers | ✅ Achieved (6 types) |
| Adaptive behavior | Changes selection based on user history | ❌ Not achieved |
| Session planning | Proposes and executes a learning plan | ❌ Not achieved |
| Autonomous initiative | ≥3 unprompted actions per session | ❌ Not achieved |

**3 of 6 metrics met.** The three that were missed are all about *agency* — the thing this project was supposed to explore.

### Honest Assessment

**What went well:**
- Data engineering was strong — card data, meta stats, decklists all working
- Test coverage is excellent for a 20-min/day side project
- Adding meta awareness (Component 7) mid-cycle was a good call — it differentiates the tool
- Code is clean and maintainable
- Shipped to its own repo with proper packaging (uv, pyproject.toml)

**What didn't happen:**
- The "Run & Iterate" Friday phase was never completed — never actually played through a full quiz session to evaluate quality
- Components 4, 5, 6 (the agentic parts) were never built
- The agent isn't really an *agent* — it's a command-driven quiz tool. User types "quiz", gets a question, types answer, gets feedback. No initiative, no planning, no memory.
- The weekly self-check ("Do I actually want to use this?") was never done
- The agentic-ness test ("Count self-initiated vs user-prompted actions") would score 0% — all actions are user-prompted

**Root cause:** The MVP was redefined on Mar 5 to include Component 7 (meta data) but that consumed the build time that might have gone to Components 4-6. The build day was a productive burst (all code + tests in one day), but it optimized for breadth (more data, more quiz types) over depth (agent behavior).

### What Was Learned

1. **MS Agents SDK ActivityHandler pattern** — understood the message → handle → respond cycle
2. **NetrunnerDB API** — v2 is stable, v3 preview is broken, decklists API has tournament badges
3. **SQLite as a cache layer** — WAL journaling, proper indexes, works well for this scale
4. **Meta integration** — tournament popularity data adds real value to quiz questions
5. **Agent ≠ tool** — Building a tool that responds to commands is different from building an agent that takes initiative. This project built a tool.

### Transferable Artifacts

- Card data fetching/caching pattern (reusable for any API → SQLite pipeline)
- Quiz question generation with fuzzy answer matching
- Test fixture patterns (in-memory SQLite with deterministic sample data)
- The meta-runner repo itself (could be resumed or handed off)

## Decision: Cycle Forward

Meta-runner achieved its MVP (Components 1+2+3+7) but didn't reach the agentic behavior that was the primary learning goal. The honest assessment: it's a decent quiz tool, not an agent.

Options considered:
1. **Continue meta-runner** — build Components 4-6 to make it truly agentic
2. **Start a new cycle** — take the learnings and explore a different idea
3. **Hybrid** — new cycle, but informed by what meta-runner revealed about agent building

**Decision:** Start a new loonshot cycle. The main value was learning, and the biggest learning is that I need to focus on agent *behavior* (planning, initiative, memory) earlier, not just agent *infrastructure* (SDK, data, tools).
