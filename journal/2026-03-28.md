## 1. What Data Is Available

### NetrunnerDB API v2 (Current, Stable)

All public, no auth required, JSON, CORS-enabled. HTTP cache support.

| Endpoint | Data | Volume (est.) |
|----------|------|---------------|
| `/public/cards` | All cards: title, text, type, faction, influence, keywords, pack | ~2,500 cards |
| `/public/card/{code}` | Single card with full attributes | — |
| `/public/decklists/by_date/{date}` | Published decklists for a given date | ~5-20/day |
| `/public/decklist/{id}` | Single decklist: name, description (HTML), author, cards `{code: count}`, mwl_code, dates | — |
| `/public/reviews` | **All reviews at once** — card title, user, ruling text (rich markdown), votes, comments with timestamps | ~hundreds of reviews, many with multi-paragraph analysis |
| `/public/rulings` | Official card rulings, NSG-verified flags | ~hundreds |
| `/public/mwl` | Full MWL/ban history: versioned card restrictions over time | ~15+ versions |
| `/public/packs`, `/public/cycles` | Card set organization | ~40 packs, ~10 cycles |
| `/public/factions`, `/public/sides`, `/public/types` | Card taxonomy | ~10 factions, 2 sides, ~15 types |

**Key observations:**
- Reviews are the richest text source — users write detailed analysis of card synergies, archetypes, meta positioning, and matchup advice. Comments add debate and corrections.
- Decklists include user-written descriptions with strategy discussion, matchup analysis, combo explanations, and meta commentary. Tournament-winning decks often have long write-ups (see the "Trains Rights" PD deck — 2000+ words of strategy).
- The `/decklists/by_date` endpoint enables incremental ingestion — poll daily for new lists.
- MWL data is versioned with dates — provides a **timeline of format changes** for tracking meta shifts.

### NetrunnerDB API v3 (Preview, In Development)

Available at `api-preview.netrunnerdb.com`. JSON:API format with richer structure:
- Cards separated from printings (no duplication)
- Format and rotation built in (`format_ids`, `card_pool_ids`, `snapshot_ids`)
- Reviews and rulings linked via relationships
- Built-in search (`filter[search]=`)
- Restriction data embedded in card attributes

V3 is more graph-friendly from the start — entities are already relationally linked. But it's preview/unstable. **Recommendation:** Design the graph schema against v3's data model but ingest from v2 (which is stable). Migrate to v3 when it's released.

### AlwaysBeRunning.net (ABR) API

Public JSON, no auth. Two key endpoints:

| Endpoint | Data |
|----------|------|
| `GET /api/tournaments` | All events: format, cardpool, date, player count, type, location, winner identities |
| `GET /api/entries?id={id}` | Per-tournament standings: player, rank, runner/corp identity+faction, NRDB deck URLs |

ABR provides what NRDB alone cannot: **performance data**. "PD went 4-1 across 3 events this month" is meta intelligence. Tournament entries link to NRDB decklists via URL, bridging both data sources.

## 2. Proposed Knowledge Graph Schema

### Node Types

```
(:Card {code, title, text, type, faction, side, influence, cost, strength, 
        memory, keywords, deck_limit, unique})

(:Decklist {id, uuid, name, description, date_created, date_updated, 
            mwl_code, tournament_badge})

(:User {id, name})

(:Review {id, text, votes, date_created, date_updated})

(:Comment {id, text, date_created, user})

(:Ruling {text, date_updated, nsg_verified})

(:Pack {code, name, date_release, cycle_code})

(:Cycle {code, name})

(:Faction {code, name, side})

(:CardType {code, name})

(:MWL {code, name, date_start, active})

(:Tournament {id, title, date, format, cardpool, player_count, type, location})

(:TournamentEntry {rank_swiss, rank_top})

(:Archetype {name, side, description})  -- derived/computed

(:Format {name})  -- standard, startup, eternal
```

### Edge Types (Relationships)

```
-- Core card-deck relationship
(:Card)-[:IN_DECK {copies: int}]->(:Decklist)

-- Authorship
(:User)-[:AUTHORED]->(:Decklist)
(:User)-[:WROTE]->(:Review)
(:User)-[:COMMENTED]->(:Comment)

-- Review/commentary structure
(:Review)-[:REVIEWS]->(:Card)
(:Comment)-[:ON]->(:Review)
(:Review)-[:MENTIONS]->(:Card)     -- extracted from card links in review text
(:Decklist)-[:MENTIONS]->(:Card)   -- extracted from card links in description

-- Identity relationship (special card type)
(:Decklist)-[:USES_IDENTITY]->(:Card)

-- Card taxonomy
(:Card)-[:BELONGS_TO]->(:Pack)
(:Pack)-[:IN_CYCLE]->(:Cycle)
(:Card)-[:HAS_FACTION]->(:Faction)
(:Card)-[:HAS_TYPE]->(:CardType)

-- Format legality (time-aware)
(:Card)-[:STATUS_IN {status: 'legal'|'banned'|'restricted', 
                      penalty: int, effective_date: date}]->(:MWL)

-- Synergy (computed from co-occurrence)
(:Card)-[:SYNERGIZES_WITH {pmi: float, shared_decks: int, 
                            archetype: string}]->(:Card)

-- Counters (derived from reviews/descriptions)
(:Card)-[:COUNTERS]->(:Card)

-- Tournament performance
(:Tournament)-[:HAS_ENTRY]->(:TournamentEntry)
(:TournamentEntry)-[:PLAYED_RUNNER]->(:Card)  -- identity
(:TournamentEntry)-[:PLAYED_CORP]->(:Card)    -- identity
(:TournamentEntry)-[:USED_DECKLIST]->(:Decklist)

-- Archetype classification (computed)
(:Decklist)-[:IS_ARCHETYPE]->(:Archetype)
(:Card)-[:STAPLE_IN]->(:Archetype)
(:Card)-[:SIGNATURE_OF]->(:Archetype)
```

### Why a Graph?

The central question: **why not just extend the SQLite schema meta-runner v1 already has?**

| Need | SQL | Graph |
|------|-----|-------|
| "What decks contain Card X?" | JOIN through junction table — fine | Direct traversal — fine |
| "Cards that co-occur with X in glacier PD decks" | Multi-table JOIN with archetype filter — awkward | 2-hop traversal with edge filter — natural |
| "What does the community say about Card X in the context of Deck Y?" | Requires joining reviews, comments, decklists, extracting mentions — painful | `(Card)-[:REVIEWED_IN]->(Review)-[:MENTIONS]->(Card)` — direct |
| "Trace why Card X fell out of meta" | Temporal JOINs across MWL versions, decklist dates, tournament results — complex query | Time-filtered traversal: `(Card)-[:IN_DECK]->(:Decklist {date > X})-[:PLACED_AT]->(:Tournament)` |
| "Cards 2 hops from X in the synergy graph" | Recursive CTE, gets ugly fast | `MATCH (c:Card)-[:SYNERGIZES*1..2]-(other)` — native |
| RAG retrieval for "how to play against glacier" | Keyword search only | Graph-augmented retrieval: find Archetype → traverse to Cards → pull Reviews + Deck descriptions |

**Verdict:** For flat card lookups and basic stats, SQL is fine (and meta-runner v1 proves it). But the moment you want to answer **relational** questions — "what works with what, in what context, according to whom, and how has that changed" — a graph is the natural representation. These are exactly the questions an agentic deck builder or meta Q&A agent needs to answer.

## 3. Graph Database Options

| Option | Pros | Cons | Fit |
|--------|------|------|-----|
| **Neo4j Community** | Industry standard, Cypher query language, great tooling, free CE, Python driver (`neo4j`), vector index for RAG | JVM overhead, heavier infra, overkill for ~2,500 cards + ~5,000 decklists | ★★★★ Best for production RAG |
| **Neo4j Aura Free** | Managed, zero-ops, same Cypher | 200K nodes/400K relationships limit (plenty), free tier may change | ★★★★ Best for getting started |
| **Memgraph** | Compatible with Cypher, in-memory (fast), Docker-native, Python driver | Smaller community, less RAG ecosystem | ★★★ Good alternative |
| **Kuzu** | Embedded graph DB (like SQLite for graphs), Python-native, no server needed, fast | Newer, smaller ecosystem, no built-in vector index | ★★★ Best for embedding in meta-runner |
| **NetworkX** | Pure Python, no infrastructure, familiar | In-memory only, no persistence, no query language, doesn't scale | ★★ Prototype only |
| **SQLite + recursive CTEs** | Already in meta-runner v1, no new dependency | Graph queries are painful, no native traversal, no vector index | ★ Don't |

**Recommendation:** Start with **Kuzu** for development (embedded, zero infra, Python-native, fast graph queries). If RAG demands vector search, graduate to **Neo4j** (which has native vector indexes for hybrid graph+vector retrieval). Kuzu's query language is also Cypher-compatible, so migration is straightforward.

## 4. Ingestion Pipeline Design

### Phase 1: Bootstrap (One-Time Full Load)

```
1. GET /api/2.0/public/cards           → CREATE (:Card) nodes
2. GET /api/2.0/public/packs           → CREATE (:Pack) nodes + relationships
3. GET /api/2.0/public/cycles          → CREATE (:Cycle) nodes + relationships
4. GET /api/2.0/public/factions        → CREATE (:Faction) nodes + relationships
5. GET /api/2.0/public/types           → CREATE (:CardType) nodes + relationships  
6. GET /api/2.0/public/reviews         → CREATE (:Review), (:Comment) nodes
                                         Parse card mentions from HTML links
7. GET /api/2.0/public/rulings         → CREATE (:Ruling) nodes
8. GET /api/2.0/public/mwl             → CREATE (:MWL) nodes + card status edges
9. Backfill decklists                  → Iterate dates, CREATE (:Decklist) nodes
                                         Parse cards_json → (:Card)-[:IN_DECK]->(:Decklist)
                                         Extract identity → (:Decklist)-[:USES_IDENTITY]->(:Card)
10. GET ABR /api/tournaments            → CREATE (:Tournament) nodes
11. GET ABR /api/entries?id={id}        → CREATE (:TournamentEntry) nodes + links
```

**Estimated volume:**
- ~2,500 Card nodes
- ~10,000-50,000 Decklist nodes (depending on how far back we go)
- ~500-1,000 Review nodes with comments
- ~500 Ruling nodes
- ~500 Tournament nodes
- ~5,000-20,000 TournamentEntry nodes
- ~200,000-500,000 IN_DECK edges (the bulk of the graph)
- ~50,000-200,000 SYNERGIZES_WITH edges (computed from co-occurrence)

This is a small graph by any standard. Even Kuzu handles this trivially.

### Phase 2: Computed Relationships

After raw data is loaded:

1. **Co-occurrence / PMI** — For each pair of cards appearing together in decklists, compute PMI. Store as `SYNERGIZES_WITH` edges above a threshold.
2. **Archetype classification** — Heuristic rules (identity + signature cards) → label decklists with archetypes. Optionally use LLM to extract archetype from descriptions.
3. **Card mention extraction** — Parse HTML card links from reviews and deck descriptions to create `MENTIONS` edges.
4. **Counter relationships** — Extract from review text where users explicitly discuss counters ("use X against Y").

### Phase 3: Incremental Updates

```
Daily cron:
  1. GET /decklists/by_date/{today}  → Upsert new decklists
  2. GET ABR /tournaments?concluded=1 → Check for newly concluded events
  3. Recompute PMI for affected cards (incremental update)
  
Weekly:
  4. GET /reviews → Diff against stored reviews (check date_update)
  5. GET /mwl → Check for new MWL entries
  6. Recompute archetype labels for new decklists
```

### Phase 4: Meta History (Time-Aware Graph)

This is the "how a card was good in one meta but not another" requirement.

**Approach: Temporal snapshots via MWL-versioned subgraphs.**

Each MWL version defines a meta era. For each era:
- Count decklists published during that era containing each card → `popularity_in_era`
- Count tournament placements → `performance_in_era`
- Store as time-stamped properties on edges:

```cypher
(:Card)-[:META_PRESENCE {
  mwl_era: "standard-ban-list-26-03",
  date_range: [2026-03-01, null],
  deck_count: 87,
  win_rate: 0.62,
  top_cut_appearances: 12
}]->(:Format)
```

This enables queries like: "Show me Card X's popularity curve across all MWL eras" — which is a simple edge scan, not a complex temporal JOIN.

**Alternative:** Use a separate `:MetaSnapshot` node per era, linking cards with their stats for that period. Cleaner for large-scale temporal analysis.

## 5. RAG Integration for Agentic Applications

### How the Graph Serves RAG

Traditional RAG: embed text chunks → vector search → retrieve top-k → feed to LLM.

Graph-augmented RAG: query the graph for structured context → combine with text retrieval → feed to LLM.

**For meta-runner v2 / deck building agent, graph-augmented RAG is significantly better:**

| Query | Pure Vector RAG | Graph-Augmented RAG |
|-------|----------------|---------------------|
| "What's good against glacier?" | Returns random chunks mentioning "glacier" | Traverses `(:Archetype {name: 'glacier'})<-[:IS_ARCHETYPE]-(:Decklist)-[:IN_DECK]->(:Card)` → finds actual glacier cards → finds `COUNTERS` edges → pulls relevant reviews |
| "Build me a criminal econ denial deck" | Keyword matching, misses structure | Finds Criminal faction → econ denial archetype → signature cards → checks format legality → pulls synergizing cards via PMI → grounds in tournament performance data |
| "Why did Endurance fall out of the meta?" | Might find a review mentioning it | Traces `(:Card {title:'Endurance'})-[:STATUS_IN]->(:MWL)` → finds ban → shows deck count drop → surfaces community discussion from reviews |

### Implementation Path

1. **Kuzu as embedded graph** — queries return structured context
2. **Card reviews + deck descriptions** — stored as text properties, embeddings generated for vector search
3. **Hybrid retrieval** — graph traversal for structure + vector search for natural language → merged context → LLM prompt
4. **Agent tools** — expose graph queries as tools the agent can call:
   - `find_synergies(card_code)` → traverses SYNERGIZES_WITH
   - `get_archetype_cards(archetype_name)` → traverses IS_ARCHETYPE → IN_DECK
   - `check_legality(card_code, format)` → traverses STATUS_IN → MWL
   - `get_meta_history(card_code)` → returns temporal popularity data
   - `search_community_knowledge(query)` → vector search over reviews

## 6. Feasibility Assessment

### ✅ Clearly Feasible

- **Data access:** All needed APIs are public, no auth required, well-documented, JSON. Both NRDB and ABR have stable APIs used by the community.
- **Graph size:** Small (~2,500 cards, ~10K-50K decklists, ~500K edges). Any graph DB handles this trivially. Even NetworkX could hold it in memory.
- **Ingestion:** Straightforward HTTP → parse → upsert pipeline. NRDB supports HTTP caching. ABR has no rate limit documented. Incremental updates via date-based endpoints.
- **Schema design:** The data is naturally a graph. Cards, decks, users, reviews, tournaments — all entities with rich relationships. The schema almost designs itself.
- **Tooling:** Python drivers exist for all recommended graph DBs. Kuzu has excellent Python integration. Neo4j has the most mature RAG ecosystem.

### ⚠️ Requires Work But Doable

- **Archetype classification:** Heuristic rules get ~70% accuracy. LLM extraction from descriptions is higher but requires API calls during ingestion. Start with heuristics, refine later.
- **Card mention extraction from HTML:** Reviews and deck descriptions contain HTML card links (`/en/card/01001`). Parsing is straightforward but needs careful HTML handling.
- **Counter/synergy extraction from text:** PMI handles statistical synergy. Semantic counters ("use X against Y") require NLP or LLM extraction from reviews. Medium effort.
- **Temporal meta tracking:** Requires defining "eras" (MWL versions work well) and computing per-era stats. The data is there; the computation is straightforward but needs design.
- **Backfilling decklists:** No `/public/decklists/all` endpoint. Must iterate by date. Could take a while but it's a one-time cost. Could also use NRDB's data export if available.

### ⛔ Risks / Concerns

- **NRDB API stability:** V2 is stable but V3 is coming. If V2 is deprecated before V3 stabilizes, there's a migration risk. Mitigated by the fact that V3 is even more graph-friendly.
- **Deck description quality:** Many decklists have empty or minimal descriptions (`"\n"`). Only ~20-30% have substantive write-ups. Tournament-winning decks tend to have better descriptions. Reviews are consistently higher quality.
- **Infrastructure creep for meta-runner v2:** This is the same trap called out in the scope doc. The graph DB is Track B infrastructure. Building it before the agent has memory/planning/initiative (Track A) reverses the priority. See Section 7.
- **Rate limiting / politeness:** No documented rate limits on NRDB, but heavy scraping without caching would be impolite. Implement HTTP caching headers and respect `last-modified`.

## 7. Fit with Meta-Runner v2 — The Honest Assessment

The scope doc (2026-03-10) explicitly warned about this:

> "This scoping document describes significant new infrastructure — new tables, new APIs, new data sources, new quiz types. That's the exact trap Cycle 1 fell into."

A knowledge graph is **even more infrastructure** than what the scope doc proposed. It's a new database engine, a new ingestion pipeline, a new query layer, and a new retrieval system.

**Where it fits in the track model:**

| Track | Focus | Graph DB Role |
|-------|-------|---------------|
| **Track A: Agent Behavior** (sessions 1-3) | SM-2 memory, session planning, initiative | **Not needed.** Agent behavior works on v1's flat data. |
| **Track B: Meta Enrichment** (sessions 4+) | Richer data, archetypes, synergies | **Natural home.** Graph replaces the proposed SQL enrichment tables. |
| **Track C: Agentic RAG** (future cycle?) | Deck building assistant, meta Q&A | **Primary value.** Graph-augmented RAG is the killer use case. |

**Recommendation:** Don't build this now. But **design it now** — having the schema, ingestion plan, and RAG integration strategy ready means Track B becomes "execute the plan" instead of "figure out what to build."

The honest sequence:
1. Finish Track A (sessions 1-3): SM-2, planning, initiative on flat data ← **current priority**
2. After gate check passes, build the graph (Track B): ingest NRDB + ABR, compute synergies
3. Swap meta-runner's data layer from SQLite popularity stats to graph queries
4. Add graph-augmented RAG for new quiz types and eventually deck building agent

## 8. Three-Round Analysis

### Round 1: Engineer's Take

The graph is technically clean. NRDB's data maps naturally to nodes and edges. Kuzu is the right starting point — embedded, Cypher-compatible, zero infra. Estimated build time for bootstrap ingestion: 2-3 focused sessions. The PMI synergy computation is well-understood math. Card mention extraction from HTML is regex on `/en/card/{code}` patterns. Temporal meta tracking via MWL eras is elegant. The RAG integration path (graph traversal → structured context + vector search → LLM) is proven architecture.

**Risk flagged:** Archetype classification accuracy. Heuristics are fast but noisy. LLM extraction is accurate but slow and expensive at ingestion time. Could prototype with heuristics and validate against community-labeled decks.

### Round 2: Devil's Advocate

This is a beautiful trap. You're designing data infrastructure when you haven't proven the agent can plan a session. The v1 meta-runner already has 2,460 cards and 1,245 decklists in SQLite — enough data to build and test all three agent behaviors. 

The graph DB doesn't make the SM-2 algorithm better. It doesn't make session planning work. It doesn't give the agent initiative. It makes the *data richer*, which makes the agent *smarter* — but only if the agent already has the behavioral capabilities to use richer data.

The "I'll just design it now" rationalization is how infrastructure projects start. You design it, then you "just want to validate the schema," then you "just need to test ingestion," and suddenly you've spent three sessions building a data pipeline instead of making the agent remember what you got wrong last session.

Also: Kuzu is a bet on an early-stage embedded graph DB. Neo4j is proven but heavy. Are you sure you want a graph query engine as a dependency for a quiz app? The meta-runner v1 SQLite schema with a proper `deck_cards` junction table and PMI computation gets you 80% of the synergy value with 0 new dependencies.

### Round 3: Decision Advisor

Both rounds are right. The engineer's graph design is sound — it's genuinely the right data model for this domain. The devil's advocate is right that building it now is a distraction.

**Verdict: Design now, build in Track B or Cycle 3.**

Concrete actions:
1. ✅ **Done today:** Full schema design, API analysis, feasibility study, RAG integration path (this document)
2. **Track A (now):** Continue with Wednesday decomposition for SM-2/planning/initiative. Use v1's flat data.
3. **Track B gate check:** After session 3, if agent behaviors work, the graph becomes the data enrichment layer. Execute the ingestion plan from this document.
4. **If Track A fails gate check:** The graph won't save you. Switch projects per the framework.

One concession to the engineer: it's worth spending 30 minutes prototyping the Kuzu schema with a small subset of cards and decklists — just to validate the data model is sound before committing to it for Track B. This is a spike, not a project.
