## Barriers Analysis

### Barrier 1: How to get the updated card pool

**What makes it hard:** Need a reliable, structured data source for Netrunner cards with format legality info.

**If removed:** Could immediately build quiz logic without worrying about data sourcing.

**Research:**
- **NetrunnerDB v3 API** (preview): `api-preview.netrunnerdb.com` — public, free, no auth needed for read
- Cards have `format_ids` (e.g., `["standard", "eternal"]`) and `card_pool_ids` baked in
- Can filter by format: `/api/v3/public/cards?filter[format_id]=standard`
- Rich data: card text, type, faction, subtypes, cost, strength, influence, images
- JSON:API format, paginated
- v2 API still works too, simpler but less format-aware

**Verdict:** ✅ Low barrier — NetrunnerDB v3 API covers this well.

---

### Barrier 2: How to store/cache it reasonably

**What makes it hard:** Don't want to hit the API on every interaction. Need local storage that's fast and queryable.

**If removed:** Could always query live and not worry about storage layer.

**Research:**
- NetrunnerDB v2 supports HTTP caching (last-modified, max-age) — v3 likely similar
- Common approaches: SQLite local DB, JSON file cache, or in-memory dict
- Card data is relatively small (~2000 cards total, maybe ~500 in standard) — fits easily in memory or a single JSON file
- SQLite gives queryability (filter by faction, type, etc.) with zero infrastructure

**Verdict:** ✅ Low barrier — SQLite or JSON file cache. Refresh on startup or daily.

---

### Barrier 3: How to keep the card pool up to date

**What makes it hard:** Card rotations happen periodically (new sets release, old ones rotate out). Stale data = wrong quizzes.

**If removed:** Could build once and never maintain.

**Research:**
- NetrunnerDB cards have `updated_at` timestamps — can do incremental sync
- Standard rotation happens ~2-3 times per year (not frequent)
- Could check API on startup, compare timestamps, refresh if stale
- Alternatively: manual refresh command ("update cards now")

**Verdict:** ✅ Low barrier — Check on startup + manual refresh. Rotation is infrequent.

---

### Barrier 4: How to come up with a usable UI that's easy to navigate

**What makes it hard:** Agent UX is conversational, but quiz/flashcard UX benefits from structure (show card, choices, score). Need to balance chat-style interaction with focused learning flow.

**If removed:** Could focus purely on agent logic without worrying about presentation.

**Research:**
- MS Agents SDK supports multiple channels: CLI, Webchat, Teams, Discord
- Webchat has Adaptive Cards — structured UI elements (images, buttons, text) inside chat
- CLI is simplest starting point (text-based quiz flow)
- Discord also supports embeds and buttons for richer interaction
- Key UX decisions: text-only quiz? show card images? multiple choice vs free recall?

**Verdict:** ⚠️ Medium barrier — Start CLI, but need to think about quiz UX patterns early so they translate to richer channels later.

---

### Barrier 5: How to keep track of personal history, learning records, preferences

**What makes it hard:** Agent needs persistent per-user state across sessions. What cards have I seen? What do I get wrong? What do I want to focus on?

**If removed:** Every session would be fresh with no memory — fine for casual use but bad for spaced repetition.

**Research:**
- MS Agents SDK has built-in state management (`UserState`, `ConversationState`)
- Storage options: in-memory (dev), Azure Blob, CosmosDB (production)
- For learning tracking specifically: need a simple model (card_id, times_seen, times_correct, last_seen, next_review)
- Spaced repetition algorithms (SM-2, Leitner) are well-documented and simple to implement
- SQLite works fine for single-user local; CosmosDB for multi-user

**Verdict:** ⚠️ Medium barrier — SDK has state primitives, but need to design the learning model. Start with SQLite locally.

---

### Barrier 6: How to make this agentic app simple enough but achieve everything

**What makes it hard:** Scope creep risk. "Simple quiz agent" can balloon into full learning platform. Need to define MVP clearly.

**If removed:** Could build the perfect app without time constraints.

**Research:**
- Framework says: decompose into parallel-explorable components (Wednesday's job)
- Agent pattern helps: start with one tool (quiz), add tools incrementally (review history, card lookup, deck analysis)
- MS Agents SDK's activity handler pattern naturally separates concerns
- Key: define what "done" looks like for a first working version

**Verdict:** ⚠️ Medium barrier — This is a discipline/scoping problem. MVP = quiz on standard cards + track right/wrong. Everything else is later.

---

### Barrier 7: How to make this available to friends (e.g., Discord)

**What makes it hard:** Going from local CLI to a deployed multi-user bot requires infrastructure, auth, hosting, and channel integration.

**If removed:** Could share immediately without deployment work.

**Research:**
- MS Agents SDK is built for multi-channel: Teams, Webchat, **Discord** (via adapter/webhook)
- Azure Bot Service handles channel registration and routing
- Deployment: Azure App Service or Container Apps
- Discord specifically: can use Direct Line or build a Discord adapter
- Need Azure subscription for Bot Service registration
- Multi-user means shared state → need CosmosDB or similar instead of local SQLite

**Verdict:** 🔴 Higher barrier — Requires Azure infra, bot registration, channel config. Save for Phase 2 after local agent works.

## Wednesday: Decompose & Measure

### UI Decision

**CLI only.** Simplest frontend that still exercises the Agents SDK meaningfully (conversation state, turn management, activity routing). The exploration goal is agentic behavior, not UI polish.

### Components (ordered by dependency, parallelizable where noted)

#### Component 1: Card Data Layer
**What:** Fetch and cache Netrunner standard card pool from NetrunnerDB v3 API into local SQLite.
**Parallel:** Yes — independent of agent work.
**Done when:** Can query cards by faction, type, side, and get full card text + images locally.

#### Component 2: Agent Skeleton (MS Agents SDK + Python)
**What:** Minimal working agent using `microsoft-agents-hosting-aiohttp` that handles a conversation turn (receive message → respond). CLI-based.
**Parallel:** Yes — independent of card data.
**Done when:** Can run `python main.py`, type a message, get a response.

#### Component 3: Quiz Tool
**What:** Agent tool that generates a quiz question from the card pool. Agent *decides* when and how to quiz (not just responding to "quiz me").
**Agentic aspect:** Agent selects card based on strategy, not random. Agent chooses question format (name→ability, ability→name, faction identification, etc.)
**Depends on:** Components 1 + 2.
**Done when:** Agent can autonomously generate and evaluate quiz questions.

#### Component 4: Learning Memory
**What:** Per-user state tracking: cards seen, correct/incorrect, last reviewed, confidence score. Stored in SQLite via Agents SDK state management.
**Agentic aspect:** Agent reads history to *plan* sessions — "You haven't seen Jinteki ice in a while" or "You keep getting tag punishment cards wrong."
**Depends on:** Components 2 + 3.
**Done when:** Agent adapts quiz selection based on user's history without being asked.

#### Component 5: Session Planner
**What:** Agent behavior that plans a coherent learning session rather than one-off questions. Decides focus areas, difficulty progression, when to review vs introduce new cards.
**Agentic aspect:** This is the core "initiative" — agent has a *goal* (improve user's card knowledge) and *autonomously sequences actions* toward it.
**Depends on:** Components 3 + 4.
**Done when:** Agent opens a session with a plan ("Today we'll cover 5 new Weyland operations and review 3 you missed last time") and executes it.

#### Component 6: Card Lookup Tool
**What:** Agent tool for on-demand card information. User or agent can trigger — "What does Endurance do?" or agent proactively explains a card after a wrong answer.
**Agentic aspect:** Agent decides *when* to explain (after wrong answer? before quiz? as a hint?) — not just when asked.
**Depends on:** Components 1 + 2.
**Done when:** Agent provides contextual card explanations as part of its autonomous behavior.

### Dependency Graph

```
[1: Card Data] ──┬──→ [3: Quiz Tool] ──┬──→ [5: Session Planner]
                 │                      │
[2: Agent Skeleton]─┬──→               [4: Learning Memory]─┘
                    │
                    └──→ [6: Card Lookup]
```

Components 1 and 2 are fully parallel. Components 3 and 6 can start once both are done. Component 4 builds on 3. Component 5 is the capstone.

### Success Metrics (SMART)

| Metric | Target | How to Measure |
|--------|--------|----------------|
| Agent runs locally | Agent starts, accepts input, responds via CLI | Manual test |
| Card data loaded | ≥90% of current standard pool cached locally | Count query vs API count |
| Quiz works | Agent generates valid quiz questions with correct answers | Manual test, 10 questions |
| Adaptive behavior | Agent changes card selection based on user history | Run 20 questions, verify wrong-answer cards reappear more frequently |
| Session planning | Agent proposes and executes a multi-step learning plan | Agent opens with a plan and follows through without prompting |
| Autonomous initiative | Agent takes ≥3 unprompted actions per session (hints, focus suggestions, review triggers) | Count during a 10-minute session |

### MVP Definition

**MVP = Components 1 + 2 + 3 only.** A CLI agent that can quiz you on standard Netrunner cards using the Agents SDK. No memory, no planning, no multi-channel. Just: agent picks a card, asks a question, evaluates your answer.

**Post-MVP layers (in order):**
1. Learning memory (Component 4)
2. Card lookup tool (Component 6)
3. Session planner (Component 5)
4. Discord frontend (future cycle)

### Evaluation Strategy

- **Weekly self-check:** Do I actually want to use this to learn cards? If not, the UX is wrong.
- **Agentic-ness test:** Count how many agent actions are self-initiated vs user-prompted. Target: >50% self-initiated by Component 5.
- **Learning effectiveness:** After 1 week of use, can I name more cards from the standard pool than before? (Informal, not rigorous.)
